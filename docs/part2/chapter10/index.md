# 第 10 章 分层记忆架构——让 Agent 拥有持久且有序的知识

在上一章中，我们看到了 GA 的工具层设计如何通过"最小原子工具集"控制信息密度。但工具只能管理单轮的输入输出，更大的挑战在于**跨任务的知识积累**——记忆。

一个只有短期记忆的 Agent，每次启动都从零开始，重复犯同样的错误，重复探索同样的路径。而一个拥有良好记忆系统的 Agent，能够把过去的经验转化为未来的效率。那么，GA 如何在"记住更多"和"上下文不爆炸"之间取得平衡？

---

## 10.1 为什么 Full-History Append 行不通

### 10.1.1 直觉理解

想象你有一个笔记本，从工作第一天起，你就把每天的所有工作内容——会议记录、邮件、临时想法、代办事项——全部按时间顺序写进去，从不整理、从不删除。

10 年后，你需要查找一条关于某个客户的关键信息。你打开这本笔记，面对的是几万页密密麻麻的文字。即使信息确实存在其中，要找到它的成本也是巨大的——你可能需要翻阅数百页才能定位到那一行关键记录。

这就是**全量历史追加（Full-History Append）** 策略的困境：记住了一切，却等于什么都没记住。

### 10.1.2 深入理解：信息密度的恶性循环

从第 7 章的信息密度视角来看，Full-History Append 会导致一个**恶性循环**：

1. 每次任务产生新的对话记录 → 追加到历史中
2. 历史越来越长 → 上下文中"与当前任务无关的旧信息"比例不断增大
3. 信息密度持续下降 → 模型的决策质量下降
4. 决策质量下降 → 任务需要更多轮次才能完成
5. 更多轮次 → 产生更多对话记录 → 回到第 1 步

实验数据直观地展示了这个问题。正如我们在第 7 章中看到的，在安装了 20 个技能并经过大量使用之后，不同 Agent 系统对一条最简请求（"Hello"）产生的**完整提示词长度（Full Prompt Length）** 差异巨大 [1]：

| 系统 | 完整提示词长度（tokens） |
|------|----------------------|
| GA | **2,298** |
| Claude Code | 22,821 |
| CodeX | 23,932 |
| OpenClaw | 43,321 |

GA 的提示词长度仅为 Claude Code 的 **10%**，OpenClaw 的 **5%**。这意味着当其他系统已经被历史记忆"淹没"时，GA 的上下文依然保持着极高的信息密度。

> 这个 10 倍以上的差距不是因为 GA "记住了更少的东西"，而是因为它**换了一种方式来记忆**。接下来我们就来看看这种方式是什么。

---

## 10.2 GA 的四层长期记忆

GA 将长期记忆组织为四个层次，每层有明确的定位和准入标准。我们用一个图书馆的类比来帮助理解：

### 10.2.1 直觉理解：图书馆的四个区域

想象 GA 的记忆系统是一座图书馆：

- **L1 索引层**就像图书馆的**目录检索台**——你不需要走进书架，只需查目录就能知道你要的书在哪。它很小（通常只有几百个 token），但能快速路由到任何需要的知识
- **L2 事实层**就像**百科全书区**——存放经过验证的、长期稳定的事实。比如"服务器 IP 是 192.168.1.100"或"项目使用 Python 3.11"。准入门槛很高，只有经过执行验证且跨任务可复用的信息才能写入
- **L3 SOP 层**就像**操作手册区**——存放可复用的操作流程。比如"如何从 HuggingFace 下载数据集"或"如何部署 Docker 容器"。每条 SOP 都来自**经行动验证的执行（Action-Verified Execution）**，而非理论推测
- **L4 归档层**就像**档案室**——存放历史执行记录的压缩存档。它不用于日常注入，平时大门紧闭。只有当需要追溯过往决策、审计历史操作时才会打开

### 4.2.2 深入理解：四层之间的路由机制

这四层记忆之间并非孤立存在，而是通过一条**L1→L2/L3 路由链**协同工作：

```
┌─────────────────────────────────────────────────────────────┐
│                      运行时上下文                             │
│  ┌──────────┐                                               │
│  │ L1 索引层 │ ←── 始终注入（紧凑指针）                       │
│  └────┬─────┘                                               │
│       │ 路由                                                 │
│       ▼                                                      │
│  ┌──────────┐  ┌──────────┐                                  │
│  │ L2 事实层 │  │ L3 SOP层 │ ←── 按需加载（file_read 触发）    │
│  └──────────┘  └──────────┘                                  │
│                                                              │
│  ┌──────────┐                                                │
│  │ L4 归档层 │ ←── 极少访问（仅追溯/审计时使用）               │
│  └──────────┘                                                │
└─────────────────────────────────────────────────────────────┘
```

**路由过程是这样的**：

1. 任务启动时，GA 只注入 L1 索引层到上下文中。L1 非常紧凑，包含高频入口、关键词映射和少量硬性约束
2. 当 LLM 在推理过程中发现需要某类知识时，它会根据 L1 中的指针，通过 `file_read` 按需加载 L2 或 L3 中的具体内容
3. 加载的内容进入当前轮的上下文，用完后随着对话历史的压缩而逐渐淡出

> 注意这里的关键设计：**路由是通过工具调用和提示策略实现的，而非硬编码的强制流水线**。模型根据任务需要自主决定是否读取更深层的记忆。这种设计既保证了灵活性，又避免了不必要的记忆加载。

### 4.2.3 每层的准入标准

四层记忆的质量靠**严格的准入标准**来保证：

**L2 事实层的准入**：
- 信息必须经过执行验证（不是"听说"，而是"确认"）
- 信息必须跨任务可复用（一次性事件不写入）
- 瞬态状态、未验证假设一律排除
- 更新采用小幅增量修改，而非大面积覆盖

**L3 SOP 层的准入**：
- 流程必须来自行动验证的执行（不是"应该这样做"，而是"这样做成功了"）
- 流程必须具有跨任务可复用性
- 只保留能直接指导错误分析或可在未来任务中复用的知识

这种严格的准入机制就是 GA 防止**记忆污染（Memory Pollution）** 的核心手段。试想，如果任何信息都可以随意写入长期记忆，那么错误的经验、过时的配置、一次性的调试信息都会混入其中，最终让记忆变得不可信。

### 4.2.4 实验印证：精炼记忆优于全量记忆

**记忆消融实验**（Memory Ablation）直接验证了"精炼记忆优于全量记忆"的设计假设 [1]。在 SOP-Bench 的危险品分类任务上：

| 记忆配置 | 记忆大小（tokens） | 任务成功率（%） |
|---------|------------------|---------------|
| 无记忆（No-Memory） | 0 | 13.87 |
| 全量记忆（Full-Memory） | 575 | 52.44 |
| 冗余记忆（Redundant-Memory） | 288 | 66.48 |
| **精炼记忆（Condensed Memory）** | **165** | **66.48** |

精炼记忆用 **165 个 token**（不到全量记忆的三分之一）达到了与冗余记忆相同的最高成功率，且远超全量记忆。更多的记忆内容并没有带来更好的效果——反而是**只保留模型尚未掌握的关键决策规则**才是最优策略。

> 这个实验结果是 GA 四层记忆设计的直接实证：不需要把所有信息都塞进记忆，经过严格筛选的精炼知识反而效果最好。

## 4.3 元记忆：记忆的管理规则

在四层记忆之上，GA 还有一个**元记忆层（Meta-Memory Layer）**——它不存储具体知识，而是定义**记忆系统本身的运行规则**。

### 4.3.1 直觉理解

如果四层记忆是图书馆的书架，那么元记忆就是图书馆的**管理章程**——它规定了：
- 图书馆有哪些区域（记忆地图）
- 每个区域收什么类型的书（准入规则）
- 新书怎么入库、旧书怎么更新（操作规范）
- 什么书不能收（排除清单）

### 4.3.2 深入理解：元记忆的三大功能

元记忆在运行时发挥三个关键作用：

**第一，提供共享参考框架。** 在任务启动时，元记忆的核心内容被注入上下文，让模型在开始执行之前就了解记忆系统的组织方式、各层的用途、以及更新的规范。这大大减少了模型的"随意写入"行为。

**第二，防止记忆污染。** 元记忆中定义了明确的更新边界：什么可以写入、什么不能写入、写入时必须遵循什么格式。比如，元记忆可能规定"L2 只能通过 `file_patch` 增量修改，禁止 `file_write` 覆盖"，从而防止一次错误的全量覆盖操作毁掉整个事实库。

**第三，维护一致性。** 当多个任务并发或顺序执行时，元记忆确保不同任务对记忆的更新不会互相冲突。它定义了"先读后写"的更新协议，要求每次写入前先读取当前状态，确认无冲突后再执行修改。这种**交叉验证（Cross-Validation）** 机制——写入前必须与现有记忆进行比对——有效防止了新写入的信息与已有事实相矛盾，或同一条知识被重复存储。

## 4.4 工作记忆：短期上下文的锚点

除了长期记忆（L1–L4）和元记忆，GA 还维护一个**工作记忆（Working Memory）**，负责在单次任务执行过程中保持上下文的连贯性。

### 4.4.1 直觉理解

工作记忆就是你桌上的**草稿纸**。当你在做一项复杂任务时，你会在草稿纸上写下当前进度、关键发现、待办事项。这张草稿纸不会被归档到图书馆——任务结束后它就可以丢掉了，除非上面有值得永久保存的内容。

### 4.4.2 深入理解：工作记忆的机制

在 GA 中，工作记忆通过 `update_working_checkpoint` 工具来维护。它的内容包括：

- **当前目标**：正在执行什么任务
- **中间结论**：已经发现了什么关键信息
- **关键约束**：需要注意什么限制条件
- **下一步行动**：接下来打算做什么

工作记忆在**每一轮对话中都会被注入**到上下文的固定位置。这意味着即使历史对话被压缩或截断（我们将在第五章详细讨论），模型仍然可以通过工作记忆快速恢复任务的关键状态。

**工作记忆与长期记忆的关键区别**：

| 维度 | 工作记忆 | 长期记忆（L1–L4） |
|------|---------|----------------|
| 生命周期 | 单次任务 | 跨任务持久 |
| 注入方式 | 每轮自动注入 | 按需加载 |
| 晋升关系 | **不自动晋升**为长期记忆 | 经过验证后才能写入 |
| 内容特征 | 当前任务的进度和发现 | 可复用的事实和流程 |

> 这里的"不自动晋升"是一个重要的设计决策。如果工作记忆自动写入长期记忆，那么大量一次性的、任务特定的信息就会污染长期记忆库。GA 要求只有经过**显式验证和蒸馏**的信息才能进入长期记忆——这个蒸馏过程将在第六章详细讨论。

### 4.4.3 三种记忆功能的分离

概括来看，GA 的记忆系统实际上维护了三种不同功能的记忆：

```
┌───────────────────────────────────────────────────────────┐
│                    GA 记忆系统全景                          │
├───────────────┬──────────────┬────────────────────────────┤
│  工作记忆       │  始终在线记忆  │  长期记忆                  │
│  (草稿纸)      │  (随身卡片)   │  (图书馆)                  │
├───────────────┼──────────────┼────────────────────────────┤
│  当前任务进度   │  工作目录      │  L2 事实 + L3 SOP + L4 归档 │
│  关键发现      │  L1 索引      │                            │
│  下一步计划    │  元记忆核心    │                            │
│               │  硬性约束      │                            │
├───────────────┼──────────────┼────────────────────────────┤
│  每轮注入      │  启动时注入    │  按需加载                   │
│               │  长任务中周期  │                             │
│               │  性刷新       │                             │
│  ~200 token   │  ~500 token  │  变量（仅加载需要的部分）      │
└───────────────┴──────────────┴────────────────────────────┘
```

> 始终在线记忆在长时间运行的任务中会**周期性刷新**，确保工作目录变更、新增约束等信息能及时反映到上下文中，而不是停留在任务启动时的快照状态。

这三种记忆的 token 开销加起来，在大多数情况下不超过 **1000 token**——这就是为什么 GA 在安装了 20 个技能后，完整提示词长度仍然只有 2,298 token 的秘密。

## 4.5 按需读取 vs 默认注入

### 4.5.1 核心策略：知道在哪，但不提前加载

GA 记忆系统最重要的设计原则可以用一句话概括：**知道知识在哪（L1 索引），但不把知识搬进来（按需加载）**。

这与大多数 Agent 系统的做法截然不同。典型的做法是在每次任务启动时，把所有"可能相关"的记忆内容全部注入上下文——这就是为什么 Claude Code 和 OpenClaw 的提示词长度高达 2 万以上。而 GA 只注入一个紧凑的索引，等到模型真正需要某条知识时，才通过工具调用去读取。

### 4.5.2 运行时的按需加载策略

**按需加载（On-Demand Loading）** 的具体流程：

1. 任务启动时，注入元记忆 + L1 索引（即"始终在线记忆"，约 500 token）
2. LLM 在推理过程中发现需要某个 SOP → 调用 `file_read` 读取对应的 L3 文件
3. SOP 内容进入当前轮上下文 → 模型根据 SOP 执行任务
4. 随着对话推进，早期加载的 SOP 内容会被第五章的压缩机制逐渐清理

这种策略确保了上下文预算始终分配给**当前任务真正需要的信息**，而不是"以防万一"预加载的大量冗余内容。

### 4.5.3 长期整合的触发提交机制

当一次任务执行产生了值得长期保存的新知识时，GA 不会立即写入记忆，而是采用**触发提交（Triggered Commit）** 机制：

1. **识别阶段**：在任务执行过程中，模型识别出潜在有价值的新信息
2. **验证阶段**：检验信息的有用性和可复用性——它是否经过了执行验证？它是否只适用于当前任务？
3. **提交阶段**：只有通过验证的信息才被写入 L2 或 L3，采用增量修改而非全量覆盖
4. **索引更新**：相应地更新 L1 索引，确保新知识可被未来任务发现

> 这种"先验证再提交"的机制，与软件工程中的版本控制理念如出一辙：你不会把未经测试的代码直接推送到主分支。同样，GA 不会把未经验证的信息直接写入长期记忆。

### 4.5.4 实验印证：长期事实记忆

分层记忆的按需加载策略不仅控制了上下文膨胀（如 §4.1 所示的提示词长度对比），还在长期事实记忆任务上表现出色。在 LoCoMo 基准测试 [2] 中，GA 在所有四个类别上都取得了最高的 F1 和 BLEU-1 分数——**无需任何向量数据库或嵌入模型** [1]：

| 系统 | 多跳推理 F1 | 多跳推理 BLEU-1 | 时序推理 F1 | 时序推理 BLEU-1 | 开放域 F1 | 开放域 BLEU-1 | 单跳查询 F1 | 单跳查询 BLEU-1 |
|------|-----------|---------------|-----------|---------------|----------|-------------|-----------|---------------|
| Mem0（嵌入式） | 39.32 | 28.43 | 50.03 | 43.33 | 18.32 | 13.80 | 40.32 | 32.33 |
| A-MEM（嵌入式） | 29.03 | 20.48 | 46.83 | 38.84 | 13.11 | 12.94 | 44.68 | 37.01 |
| OpenClaw | 21.43 | 18.41 | 22.56 | 20.43 | 9.56 | 9.03 | 23.44 | 24.21 |
| **GA** | **43.33** | **39.96** | **52.23** | **51.11** | **20.41** | **15.31** | **45.69** | **40.66** |

这个结果表明，通过精确的记忆组织和过滤，GA 不需要额外的检索基础设施就能实现优秀的长期记忆能力。

## 4.6 本章小结

本章深入剖析了 GA 的分层记忆架构。我们看到：

1. **Full-History Append 行不通**：无限制地追加历史会导致信息密度的恶性循环
2. **四层长期记忆**（L1 索引→L2 事实→L3 SOP→L4 归档）通过严格的准入标准和分层存储，在"记住更多"和"保持精炼"之间取得了平衡
3. **元记忆**为整个记忆系统提供了自管理的规则框架
4. **工作记忆**作为短期锚点，确保单次任务内的上下文连贯性
5. **按需加载 + 触发提交**确保了上下文预算始终用在刀刃上

GA 记忆系统的核心智慧可以归结为一句话：**好的记忆不是记住一切，而是在需要的时候能找到需要的东西。**

但记忆分层解决的是"什么信息该进入上下文"的问题。当任务执行到中后期，即使有了分层记忆，单次对话内累积的历史消息依然会越来越长。GA 如何在不丢失关键信息的前提下，主动控制每一轮的上下文大小？这就是下一章的主题——**上下文截断与压缩**。
