# 第三章：最小原子工具集——用 9 个工具覆盖完整能力环

在上一章中，我们建立了 GenericAgent（GA）的第一性原理——**上下文信息密度最大化**，并了解了 LLM 在处理长上下文时面临的三重陷阱。我们还预告了 GA 通过四个核心机制来优化信息密度。本章将深入第一个机制：**最小原子工具集（Minimal Atomic Toolset）**。

一个很自然的问题是：既然 Agent 需要与外部世界交互，那为什么不给它尽可能多的工具？工具越多，能力不就越强吗？

---

## 3.1 工具膨胀的代价

### 3.1.1 直觉理解

想象你走进一家五金店。你需要拧一颗螺丝，店员递给你一本 200 页的工具目录，里面有 53 种不同的螺丝刀——十字的、一字的、六角的、带棘轮的、电动的、可弯曲的……你需要先花 10 分钟翻阅目录，理解每种螺丝刀的区别，然后才能做出选择。

而另一家店只给你一把通用螺丝刀和一个万能批头套装。你拿起来就能干活。

对于 LLM Agent 来说，"翻阅工具目录"的代价远比你想象的大。因为**每一轮对话**中，所有可用工具的定义都会被注入到上下文中。工具越多，这些定义占用的 token 就越多——而且这是**每轮都要重复支付的固定成本**。

### 3.1.2 深入理解：工具数量的三重代价

工具膨胀带来的问题不仅仅是"占用 token"这么简单，它在三个层面同时产生负面影响：

**第一，上下文预算挤占。** 在一个典型的 Agent 系统中，工具定义（Schema）包含工具名称、功能描述、参数类型和约束说明。每个工具的 Schema 通常占用 100–300 个 token。如果系统暴露 53 个工具，仅工具定义就可能消耗 5000–15000 个 token——这些 token **每轮对话都会出现**，直接挤占了留给任务信息、历史对话和记忆内容的空间。

**第二，选择困难与语义重叠。** 当多个工具的功能存在重叠时，模型需要消耗额外的推理能力来区分它们。比如，`FileReadTool` 和 `GrepTool` 都可以读取文件内容，`BashTool` 也可以通过 `cat` 命令实现同样的效果。模型需要理解这三个工具的微妙差异才能做出正确选择——而这种"选择负担"会随着工具数量的增加而急剧上升。

**第三，长尾工具的"隐形浪费"。** 在实际使用中，即使暴露了几十个工具，真正被频繁调用的往往只是少数几个。报告中的实验数据 [1] 清楚地展示了这一现象：

| Agent 系统 | 工具总数 | 最常用工具及占比 |
|-----------|---------|----------------|
| Claude Code | 53 个 | AgentTool (50.4%)、WebFetchTool (22.1%)、FileReadTool (10.6%) |
| OpenClaw | 18 个工厂 | browser (32.5%)、process (20.5%)、write (15.7%) |
| GA | 9 个 | code_run (34.4%)、web_execute_js (31.2%)、file_read (17.2%) |

我们可以看到，Claude Code 虽然暴露了 53 个工具，但超过 80% 的调用集中在仅仅 3 个工具上，剩余 47 个工具只贡献了不到 1% 的调用量。这意味着这 47 个工具的 Schema **每轮都在消耗上下文预算，却几乎从不被使用**——这正是上下文信息密度的反面。

> 回顾第一章的核心公式，工具定义中的"长尾工具"是典型的低信息密度内容：它们增大了分母（上下文总长度），却几乎不增加分子（决策相关信息量）。

那么，GA 如何解决这个问题？答案出奇地简单：**只保留 9 个工具**。

## 3.2 GA 的 9 个原子工具

### 3.2.1 直觉理解：乐高积木而非瑞士军刀

GA 的工具设计哲学可以用一个类比来概括：**不要瑞士军刀，要乐高积木**。

瑞士军刀为每种功能设计了一个专用部件——开瓶器、剪刀、锯子、锉刀……虽然功能多，但每个部件都是固定形态，无法组合。而乐高积木只有几种基本形状的砖块，却可以通过组合搭建出无限可能。

GA 的 9 个工具就是 9 块"乐高积木"——每个工具只做一件事（**单一职责，Single Responsibility**），但它们可以自由组合来完成复杂任务。

### 3.2.2 深入理解：五大能力类与九大原子工具

GA 将 Agent 与外部世界的所有交互归纳为**五大能力类（Capability Classes）**，每个能力类下放置 1–3 个原子工具：

```
┌─────────────────────────────────────────────────────────┐
│                  GA 原子工具集架构                        │
├──────────────┬──────────────────────────────────────────┤
│  能力类       │  工具                                    │
├──────────────┼──────────────────────────────────────────┤
│  文件操作     │  file_read · file_patch · file_write     │
│  代码执行     │  code_run                                │
│  网页交互     │  web_scan · web_execute_js               │
│  记忆管理     │  update_working_checkpoint               │
│              │  start_long_term_update                   │
│  人机协作     │  ask_user                                │
└──────────────┴──────────────────────────────────────────┘
```

这 9 个工具覆盖了一个完整的**感知-行动-记忆-求助循环（Observation-Action-Memory-Escalation Loop）**：

- **感知（Observation）**：`file_read` 读取文件内容，`web_scan` 扫描网页信息——让 Agent 能"看见"外部世界的状态
- **行动（Action）**：`file_patch` / `file_write` 修改文件，`code_run` 执行代码，`web_execute_js` 操作浏览器——让 Agent 能"改变"外部世界
- **记忆（Memory）**：`update_working_checkpoint` 维护短期工作记忆，`start_long_term_update` 触发长期记忆写入——让 Agent 能"记住"重要信息
- **求助（Escalation）**：`ask_user` 在需要人类决策时请求介入——让 Agent 知道自己的能力边界

> 在第一部分的应用篇中，我们已经实际使用过这些工具。本章的重点不是"怎么用"，而是理解"为什么是这 9 个，为什么不多也不少"。

### 3.2.3 关键工具的设计细节

每个原子工具都经过了面向**决策效率（Decision Efficiency）** 和**token 经济性（Token Economy）** 的专门优化。我们来看几个典型例子：

**file_read：精准读取，拒绝"全文搬运"**

`file_read` 不是简单的"把整个文件内容返回"。它支持三种读取模式：

```python
# 模式 1：指定范围读取——只读文件的第 100-120 行
file_read(path="main.py", start=100, count=20)

# 模式 2：关键词锚定——跳到包含 "def train" 的位置
file_read(path="main.py", keyword="def train")

# 模式 3：带行号输出——方便后续用 file_patch 精确编辑
file_read(path="main.py", start=1, count=50, show_linenos=True)
```

这种设计确保 Agent 只获取"当前决策需要的那几行代码"，而不是把一个 5000 行的文件全部塞入上下文。

**file_patch：唯一匹配，拒绝"沉默错误"**

`file_patch` 要求 `old_content` 参数在目标文件中**恰好匹配一处**。如果匹配零处（内容不存在）或多处（存在歧义），工具会直接报错而非静默执行。这种**快速失败（Fail-Fast）** 设计避免了"改错位置"的沉默错误——在 Agent 自主运行的场景下，沉默错误比报错更危险，因为它可能在很多步之后才被发现。

**web_scan：语义提取，拒绝"原始 DOM"**

`web_scan` 是原子工具中优化最激进的一个。它不是简单地返回网页的 HTML 源码，而是执行了一套完整的**布局分析算法（Layout Analysis Algorithm）**：

1. 克隆当前页面的 DOM 树
2. 计算每个元素的可见性（是否被遮挡、是否在视口内）
3. 将区域分类为"主要内容"和"非核心区域"（导航栏、侧边栏、广告等）
4. 移除被遮挡或隐藏的元素
5. 将结果序列化为结构化的精简文本

这套处理将 token 消耗降低了一个数量级——一个原始 DOM 可能包含数万个 token，而经过 `web_scan` 处理后通常只剩几千个 token，且关键信息完整保留 [1]。

> 这是"上下文信息密度最大化"原则在工具层的直接体现：同样是"看网页"，返回原始 HTML 是低信息密度的（大量标签、样式、脚本），返回结构化摘要是高信息密度的。

## 3.3 权限-责任对齐模型

工具不仅是"能力"的体现，也是"权限"的边界。GA 采用了一种**权限-责任对齐（Authority-Responsibility Alignment）** 的设计模式：Agent 的能力严格由注入的工具集定义，**Agent 不能自我提权（Self-Escalate）**。

### 3.3.1 直觉理解

这就好比一个公司的权限管理系统。每个员工的操作权限由管理员在系统中配置，员工不能自己给自己加权限。但管理员可以提前给某些员工配置"受控提权通道"——比如 IT 人员可以通过特定审批流程获得服务器的临时管理员权限。

在 GA 中：
- **只读工具**（`file_read`、`web_scan`）只能观察，不能修改——Agent 可以看到文件内容，但无法改变任何状态
- **编辑工具**（`file_patch`、`file_write`）可以修改文本，但受参数约束——必须精确指定修改位置和内容
- **执行工具**（`code_run`、`web_execute_js`）可以触发系统动作——这是风险最高的工具，但也是能力最强的

### 3.3.2 深入理解：`code_run` 作为受控提权点

在这个权限体系中，`code_run` 扮演着一个特殊角色——它是**受控提权点（Controlled Escalation Point）**。通过 `code_run`，Agent 可以执行任意 Python 或 Bash 代码，这意味着它理论上可以做到工具集未直接提供的几乎任何事情：安装第三方包、调用系统 API、处理复杂数据……

但 `code_run` 的自由并非无限制的：

1. **每轮只能调用一次**：确保每次执行的结果都被观察和评估后，才能决定下一步行动
2. **返回值受截断控制**：过长的输出会被截断，防止上下文爆炸
3. **运行环境受限**：在沙箱或受控环境中执行，而非直接在宿主系统上

这种设计让 `code_run` 成为了"万能胶水"——当现有的 8 个工具无法直接完成某项任务时，`code_run` 可以通过编写代码来填补能力缺口。

### 3.3.3 Schema 契约与统一分发

GA 的所有工具通过一套统一的**Schema 契约（Schema Contract）** 来定义。每个工具的定义包含三个部分：

```json
{
  "type": "function",
  "function": {
    "name": "file_read",              // 工具名称
    "description": "Read file content with optional range and keyword anchoring",  // 功能描述
    "parameters": {                   // 参数约束（JSON Schema 格式）
      "type": "object",
      "properties": {
        "path": {"type": "string"},
        "start": {"type": "integer"},
        "count": {"type": "integer"},
        "keyword": {"type": "string"}
      },
      "required": ["path"]
    }
  }
}
```

在运行时，这些 Schema 被注入到模型的上下文中。模型根据 Schema 生成结构化的**工具调用请求（Tool Call）**，然后由一个**统一分发器（Unified Dispatcher）** 将请求路由到对应的本地执行器。这种设计实现了调用语义与实现细节的分离——模型只需要知道"工具能做什么"和"参数格式是什么"，不需要知道工具的具体实现方式。

> 统一分发器还负责执行前的参数校验和执行后的结果格式化，确保所有工具的输入输出具有一致的结构，这进一步降低了模型的认知负担。

## 3.4 工具组合的威力：实验印证

理论上讲，9 个原子工具覆盖了完整的能力环。但在实践中，它们真的能替代拥有几十个专用工具的系统吗？报告中的实验数据给出了令人信服的回答 [1]。

### 3.4.1 能力覆盖：专用工具 → 原子组合

下表展示了几个具体的替代案例：

| 原系统 | 专用工具 | 功能 | GA 替代方案 |
|-------|---------|------|-----------|
| Claude Code | `GlobTool` | 文件模式匹配 | `code_run`（执行 glob 脚本） |
| Claude Code | `GrepTool` | 文本搜索定位 | `code_run` + `file_read` |
| Claude Code | `WebSearchTool` | 在线搜索 | `web_execute_js` + `web_scan` |
| Claude Code | `NotebookEditTool` | Jupyter 编辑 | `file_read` + `code_run` |
| OpenClaw | nodes（结构化数据） | CSV 统计 | `code_run` |

每个专用工具的功能都可以通过 2–3 个原子工具的组合来实现。这并非牺牲了能力，而是将"内置的专用能力"转化为"通过组合涌现的通用能力"。

### 3.4.2 效率对比：更少的工具，更低的成本

在 5 个长链复杂任务上的对比实验中 [1]：

| 指标 | Claude Code (53 工具) | GA (9 工具) | 比值 |
|------|---------------------|------------|------|
| 任务成功率 | 100% | 100% | 持平 |
| 总 Token 消耗 | 537,413 | 188,829 | GA 仅为 35.1% |
| 平均请求数 | 32.6 | 11.0 | GA 仅为 33.7% |
| 平均工具调用数 | 22.6 | 12.8 | GA 仅为 56.6% |

GA 在保持 100% 任务成功率的同时，Token 消耗降低到 Claude Code 的三分之一。这不是因为 GA 做了更少的工作，而是因为它的**上下文更干净**——更少的工具定义、更精准的信息提取、更简洁的交互轮次。

> 这些数据直观地印证了第一章的核心论点：上下文信息密度的提升直接转化为效率的提升。当上下文中每个 token 都更"有用"时，模型需要更少的轮次就能做出正确决策。

## 3.5 工具缺席时的兜底机制

在实际运行中，模型偶尔会出现"不调用任何工具"的情况——可能是输出被截断，也可能是模型陷入了纯文本推理而忘记调用工具。GA 设计了一个**无工具兜底机制（No-Tool Fallback）** 来处理这种情况：

1. **检测**：当模型的回复中不包含任何工具调用时，系统自动检测是否存在空响应、截断输出或未提交的代码块
2. **纠正**：注入一条纠正提示，引导模型重新进入结构化的工具调用循环
3. **升级**：如果连续多轮都无法恢复，系统会强制触发 `ask_user`，将控制权交还给用户

此外，GA 还实现了**基于轮次的升级策略（Turn-Based Escalation）**：周期性的警告提示强制模型切换策略，防止它在错误的方向上无限消耗轮次。一个硬性上限确保了模型不会无限运行下去。

## 3.6 本章小结

本章深入分析了 GA 最小原子工具集的设计原理。我们从"工具膨胀的代价"出发，理解了为什么"工具越多≠能力越强"——过多的工具会挤占上下文预算、增加选择负担、制造长尾浪费。

GA 的解法是保留 9 个原子工具，按照乐高积木的思路，通过组合来覆盖几乎所有任务场景。实验数据表明，这种最小化设计不仅没有牺牲任务完成能力，反而在 token 效率上实现了数倍的提升。

GA 的工具设计带给我们的核心启示是：

- **少即是多**：更少的工具 → 更小的 Schema 开销 → 更高的上下文信息密度
- **组合优于专用**：通用原子工具的组合能力 ≥ 大量专用工具的独立能力
- **约束即引导**：明确的权限边界让 Agent 更可控、更可追溯

在下一章中，我们将深入 GA 的另一个核心机制——分层记忆架构。如果说工具集解决的是"Agent 能做什么"的问题，那么记忆系统解决的是一个更根本的问题：**Agent 如何"记住"跨任务的经验，又不让过往记忆淹没当前决策？**
